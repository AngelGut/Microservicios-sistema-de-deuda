# Esquemas de Base de Datos — Debt Manager Microservices

Cada microservicio tiene su propia base de datos independiente. Ningún servicio accede directamente a la BD de otro — la comunicación siempre es por API REST.

---

## auth-service — `auth.db` (SQLite)

Gestiona los usuarios del sistema (credenciales y roles).

```sql
CREATE TABLE users (
    id         TEXT        NOT NULL PRIMARY KEY,  -- UUID generado por @PrePersist
    email      TEXT        NOT NULL UNIQUE,       -- Usado para login
    password   TEXT        NOT NULL,              -- Hash BCrypt
    role       TEXT        NOT NULL,              -- "ADMIN" | "USER"
    enabled    BOOLEAN     NOT NULL,              -- false = soft delete
    created_at TIMESTAMP   NOT NULL               -- Fecha de creación
);
```

| Campo | Descripción |
|-------|-------------|
| `id` | UUID en formato String, generado automáticamente |
| `email` | Único, usado como credencial de acceso |
| `password` | Nunca se guarda en texto plano, siempre BCrypt |
| `role` | Solo dos roles: `ADMIN` (acceso total) y `USER` (acceso limitado) |
| `enabled` | `false` equivale a eliminado (soft delete) — el registro se mantiene |

**Admin por defecto** creado al arrancar:
- email: `admin@tejada.com` / password: `Admin2026!` / role: `ADMIN`

## debtor-service — `debtor.db` (SQLite)

Almacena los deudores (personas o empresas que tienen deudas).

```sql
CREATE TABLE debtors (
    id       BIGINT      GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name     VARCHAR(120) NOT NULL,
    document VARCHAR(50)  NOT NULL UNIQUE,  -- Cédula / RNC
    email    VARCHAR(120) NOT NULL UNIQUE,
    type     VARCHAR(20)  NOT NULL          -- "PERSONA" | "EMPRESA"
);
```

| Campo | Descripción |
|-------|-------------|
| `id` | Auto-incremental numérico |
| `name` | Nombre completo o razón social |
| `document` | Cédula (persona) o RNC (empresa), único en el sistema |
| `email` | Usado por notification-service para enviar correos |
| `type` | Tipo de deudor: persona física o empresa |

## debt-service — `debt.db` (SQLite)

Almacena las deudas asignadas a cada deudor.

```sql
CREATE TABLE debts (
    id              TEXT           PRIMARY KEY,       -- UUID
    debtor_id       TEXT           NOT NULL,          -- Referencia al deudor
    description     VARCHAR(300)   NOT NULL,          -- Descripción de la deuda
    original_amount DECIMAL(15,2)  NOT NULL,          -- Monto original
    current_balance DECIMAL(15,2)  NOT NULL,          -- Saldo pendiente actual
    currency        VARCHAR(3)     NOT NULL DEFAULT 'DOP',  -- ISO 4217
    status          VARCHAR(10)    NOT NULL DEFAULT 'ACTIVA',
    due_date        DATE,                             -- Fecha de vencimiento
    created_at      TIMESTAMP      DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP      DEFAULT CURRENT_TIMESTAMP
);
```

| Campo | Descripción |
|-------|-------------|
| `id` | UUID, generado en Java |
| `debtor_id` | UUID del deudor en debtor-service (no FK física, referencia lógica) |
| `original_amount` | Monto inicial de la deuda, nunca cambia |
| `current_balance` | Saldo restante, se reduce con cada pago vía `apply-payment` |
| `currency` | Código de moneda ISO: `DOP`, `USD`, etc. |
| `status` | `ACTIVA` → en curso, `PAGADO` → saldada completamente |
| `due_date` | Fecha límite de pago, usada por notification-service para recordatorios |

## payment-service — `payment.db` (SQLite)

Registra cada pago realizado contra una deuda.

```sql
CREATE TABLE payments (
    id           INTEGER  PRIMARY KEY AUTOINCREMENT,
    debt_id      TEXT     NOT NULL,       -- UUID de la deuda en debt-service
    amount       REAL     NOT NULL,       -- Monto pagado
    payment_date TEXT     NOT NULL,       -- Fecha del pago (ISO-8601)
    note         TEXT,                   -- Nota opcional del pago
    created_at   TEXT     NOT NULL        -- Fecha de registro (ISO-8601)
);
```

| Campo | Descripción |
|-------|-------------|
| `id` | Auto-incremental numérico |
| `debt_id` | UUID de la deuda (referencia lógica, sin FK física hacia debt-service) |
| `amount` | Monto del pago parcial o total |
| `payment_date` | Cuándo se realizó el pago (no cuándo se registró) |
| `note` | Observaciones opcionales (ej: "abono mensual", "liquidación") |

**Flujo al registrar un pago:**
1. Se inserta en esta tabla
2. Se llama a `debt-service` para reducir `current_balance`
3. Si `current_balance = 0`, `debt-service` cambia `status` a `PAGADO`

## ai-risk-service — `ai-risk.db` (SQLite)

Almacena el perfil de riesgo calculado para cada cliente. Es el "caché persistente" del cálculo de riesgo.

```sql
-- V1: esquema original
-- V2: migración de client_id INTEGER → TEXT para soportar UUIDs

CREATE TABLE client_risk (
    id                  INTEGER  PRIMARY KEY AUTOINCREMENT,
    client_id           TEXT     NOT NULL UNIQUE,  -- UUID del deudor
    risk_level          TEXT     NOT NULL,          -- GOOD_CLIENT | LOW_RISK | HIGH_RISK
    risk_score          REAL     NOT NULL DEFAULT 0,
    total_days_late     INTEGER  NOT NULL DEFAULT 0,
    late_payment_count  INTEGER  NOT NULL DEFAULT 0,
    payment_count       INTEGER  NOT NULL DEFAULT 0,
    last_calculated_at  TEXT     NOT NULL,          -- ISO-8601
    created_at          TEXT     NOT NULL,
    updated_at          TEXT     NOT NULL
);
```

| Campo | Descripción |
|-------|-------------|
| `client_id` | UUID del deudor (mismo ID que en debtor-service) |
| `risk_level` | Resultado de las reglas de negocio |
| `risk_score` | Puntuación numérica 0-100 |
| `total_days_late` | Suma de todos los días de mora históricos |
| `late_payment_count` | Cuántos pagos se hicieron tarde |
| `payment_count` | Total de pagos realizados |
| `last_calculated_at` | Cuándo se calculó por última vez (máx. 24h de antigüedad) |

**Clasificación de riesgo:**
```
total_days_late = 0      → GOOD_CLIENT  (sin mora histórica)
total_days_late < 30     → LOW_RISK     (mora leve)
total_days_late >= 30    → HIGH_RISK    (mora significativa)
```

## notification-service — `notification.db` (SQLite)

Registra todas las notificaciones enviadas o intentadas. Funciona como log de auditoría y como mecanismo anti-duplicados.

```sql
CREATE TABLE notification_logs (
    id              INTEGER  PRIMARY KEY AUTOINCREMENT,
    type            TEXT     NOT NULL,   -- PAYMENT_CONFIRMATION | PAYMENT_REMINDER
    reference_id    TEXT,                -- ID de la deuda o pago relacionado
    recipient_email TEXT     NOT NULL,
    recipient_name  TEXT,
    status          TEXT     NOT NULL,   -- SENT | FAILED
    message         TEXT,               -- Detalle del resultado o error
    processed_at    TEXT     NOT NULL,   -- ISO-8601
    created_at      TEXT     NOT NULL    -- ISO-8601
);

-- Índice para verificar duplicados eficientemente
CREATE INDEX idx_notif_type_ref_status
    ON notification_logs (type, reference_id, status, processed_at);

-- Índice para búsqueda por destinatario
CREATE INDEX idx_notif_recipient
    ON notification_logs (recipient_email);
```

| Campo | Descripción |
|-------|-------------|
| `type` | `PAYMENT_CONFIRMATION` (por pago registrado) o `PAYMENT_REMINDER` (por vencimiento próximo) |
| `reference_id` | ID de la deuda o pago que originó la notificación |
| `status` | `SENT` = correo enviado exitosamente, `FAILED` = hubo un error |
| `message` | Mensaje de éxito o descripción del error para diagnóstico |

**Anti-duplicados:** antes de enviar un recordatorio, el scheduler consulta:
```sql
SELECT EXISTS (
    SELECT 1 FROM notification_logs
    WHERE type = 'PAYMENT_REMINDER'
      AND reference_id = ?
      AND status = 'SENT'
      AND processed_at >= [inicio del día actual]
)
```

## Relaciones entre Bases de Datos

Los servicios no comparten base de datos. Las referencias entre entidades son **lógicas** (UUID almacenado como texto), no físicas (sin FOREIGN KEY entre BDs).

```
debtors.id (debtor-service)
    └──► debts.debtor_id (debt-service)          referencia lógica
              └──► payments.debt_id (payment-service)     referencia lógica
                        └──► client_risk.client_id (ai-risk-service)  mismo UUID del deudor
                                  └──► notification_logs.reference_id  ID de deuda o pago
```

**Consecuencia:** si se borra un deudor en debtor-service, sus deudas en debt-service NO se borran automáticamente (no hay cascada entre BDs). La consistencia es responsabilidad de la capa de aplicación.

## Herramienta de Migraciones: Flyway

Todos los servicios usan Flyway para versionado del esquema:

```
src/main/resources/db/migration/
  V1__create_tabla_principal.sql    ← Esquema inicial
  V2__descripcion_del_cambio.sql    ← Cambio posterior
```

Flyway ejecuta las migraciones automáticamente al arrancar. Si ya fueron aplicadas, las ignora. Si el servicio se despliega en un entorno nuevo, aplica todas desde V1 en orden.
